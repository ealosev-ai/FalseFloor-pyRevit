## 🚀 Быстрый старт

### Требования

- **Revit:** 2024 или 2026
- **pyRevit:** 6.1.0.x
- **Python:** 3.9+ (встроен в pyRevit)
- **Git:** для клонирования репозитория

### Установка для разработки

1. **Клонируйте репозиторий:**

   ```bash
   git clone <repository-url>
   cd RaisedFloor.extension
   ```

2. **Установите зависимости:**

   ```bash
   pip install -r requirements.txt
   ```

3. **Добавьте расширение в pyRevit:**
   - Откройте pyRevit в Revit
   - Перейдите в `Extensions` → `Add Folder`
   - Выберите папку `RaisedFloor.extension`
   - Нажмите `Reload pyRevit`

4. **Загрузите семейства:**
   - `RF_Tile.rfa`
   - `RF_Stringer.rfa`
   - `RF_Support.rfa`

### Алгоритм подготовки (01_Setup)

#### Перекрытие (Slab)

Пользователь выбирает перекрытие (Floor) или Part, указывает базовую точку, шаг плитки (по умолчанию 600×600 мм) и высоту фальшпола. Параметры сохраняются на элементе через `RF_` shared parameters.

#### Периметр (Contour)

1. Извлечение верхней грани перекрытия (`get_top_face()`).
2. Определение внешнего контура и отверстий (колонны, проёмы).
3. На плане рисуются зелёные линии (стиль `RF_Contour`).
4. ID линий сохраняются в параметре `RF_Contour_Lines_ID`.

#### Сетка (Grid)

1. Построение прямоугольной сетки с шагом из параметров.
2. Линии обрезаются по контуру зоны (Clipper2).
3. Линии рядом с колоннами подсвечиваются синим.
4. Перестраивается при изменении смещения `RF_Shift_X` / `RF_Shift_Y`.

### Алгоритм оптимизации смещения (02_OptimizeLayout)

Файл: `lib/floor_exact.py` → `find_best_shift()`

#### Трёхфазный поиск

**Фаза 1 — Грубый поиск + snap:**

- Перебор сетки смещений с шагом 50 мм (`DEFAULT_COARSE_SHIFT_STEP_MM`).
- Добавление аналитических snap-кандидатов: `_snap_shifts_for_axis()` — по вершинам контура; `_snap_pairs_for_holes()` — по углам отверстий.
- Каждый кандидат оценивается через `evaluate_floor_shift()`.

**Фаза 2 — Уточнение:**

- Top-5 кандидатов фазы 1 (`DEFAULT_REFINE_TOP_N`).
- В радиусе ±60 мм (`DEFAULT_REFINE_RADIUS_MM`) от каждого с шагом 10 мм (`DEFAULT_REFINE_SHIFT_STEP_MM`).
- Переоценка, повторное ранжирование.

**Фаза 3 — Округление подрезок (cut-round polish):**

- Top-3 кандидата фазы 2.
- Локальное смещение на CUT_ROUND=10 мм для кратности подрезок.
- Финальное ранжирование.

#### Оценка смещения (`evaluate_floor_shift`)

Для каждого смещения (dx, dy):

1. Каждая ячейка пересекается с точным контуром зоны (Clipper2 boolean intersection).
2. Классификация ячеек: **Целая** / **Простой подрез** / **Сложный подрез** (L/T/U) / **Пустая** / **Микрофрагмент**.
3. Пороги подрезов:
   - < 50 мм — micro-fragment (удаляется)
   - < 100 мм — недопустимый подрез
   - 100–150 мм — нежелательный подрез
   - 150–200 мм — допустимый подрез
   - ≥ 200 мм — хороший подрез

#### Ранжирование

Приоритет: минимум несрезанных отверстий/колонн → минимум недопустимых и нежелательных подрезов → минимум микрофрагментов → минимум линий у кромок отверстий → технологичность (кратность CUT_ROUND_MM).

Результат: Top-10 лучших вариантов (`DEFAULT_TOP_N`) для отчёта, применяется лучший.

### Алгоритм размещения стрингеров (03_Stringers/Place)

### Общая схема

Стрингеры размещаются в два уровня: **верхние** (несут плитку) и **нижние** (перпендикулярны верхним, несут верхние). Направление задаётся параметром `RF_Direction_Axis` (X или Y).

```
Плитка → лежит на верхних стрингерах
Верхние стрингеры → лежат на нижних стрингерах
Нижние стрингеры → лежат на стойках
```

### Pipeline (шаги)

#### 1. Осевые из сетки

- **Верхние**: горизонтальные (dir_x=True) линии из `main_lines`, позиции из `main_keys`
- **Нижние**: вертикальные, полный пролёт bbox, позиции из `lower_positions` (шаг `lower_step`)

#### 2. Контурная обвязка

Обвязка строится из offset-контура зоны (Clipper2 `InflatePaths`):

- **wall** (стеновые) — из `outer_paths` (внешняя граница). Неприкосновенны.
- **hole** (вокруг отверстий) — из `hole_paths`. Могут быть отсеяны.

Разделение критично: стеновые гарантируют опору плитки у стен, hole-контурные могут дублировать осевые.

#### 3. Extend ends (нахлёст в углах)

Контурные удлиняются на ширину поперечного стрингера для перекрытия в углах.

#### 4. Clip (зазор 5 мм)

Все сегменты обрезаются по контуру зоны с зазором 5 мм (even-odd ray-casting).

#### 5. Cut — шахматный порядок стыков

**Верхние осевые** — стыки на нижних позициях:

```
lower_positions = [p0, p1, p2, p3, p4, ...]
lp_even = [p0, p2, p4]  (чётные)
lp_odd  = [p1, p3]       (нечётные)

main_keys[0] → режется по lp_even
main_keys[1] → режется по lp_odd
main_keys[2] → режется по lp_even
...
```

Стыки опёрты на нижний стрингер. Соседние ряды смещены — нет слабой линии.

**Нижние осевые** — стыки в серединах пролётов между верхними:

```
main_keys = [m0, m1, m2, m3]
mk_mids = [(m0+m1)/2, (m1+m2)/2, (m2+m3)/2]
mk_mids_even = [mid0, mid2]
mk_mids_odd  = [mid1]

lower_positions[0] → режется по mk_mids_even
lower_positions[1] → режется по mk_mids_odd
...
```

Нижний непрерывен под каждым верхним (точка макс. нагрузки). Стык — в центре пролёта (минимум нагрузки). Шахматный порядок сохраняется.

**Контурные** — wall и hole по соответствующим позициям без шахматки.

#### 5a. Drop near parallel (только нижние hole)

Если нижний осевой проходит параллельно и ближе `pw + 5 мм` к нижнему hole-контурному — контурный отсеивается. Стеновые и **все верхние** hole-контурные сохраняются (плитке нужны ≥2 опоры).

#### 6. Dedup

`grid + wall_contour + filtered_hole_contour` → дедупликация.

#### 7. Filter short

Сегменты короче `max(pw × 0.5, 30 мм)` удаляются.

### Ключевые конструктивные решения

| Решение | Почему |
|---------|--------|
| Стыки верхних на нижних | Стык опёрт — прочность |
| Стыки нижних в середине пролёта | Нижний непрерывен под верхним (макс. нагрузка) |
| Шахматка обоих уровней | Нет слабой поперечной линии |
| Wall-контурные неприкосновенны | Опора плитки у стен |
| Hole-контурные нижние отсеиваются | Дублирование осевой бессмысленно |
| Hole-контурные верхние сохраняются | Плитке нужны ≥2 параллельные опоры |

### Алгоритм размещения стоек (04_Supports/Place)

Файл: `lib/floor_common.py` → `build_support_nodes()`

1. Для каждого нижнего стрингера (сегмента):
   - Обязательные стойки на обоих концах сегмента.
   - Промежуточные стойки на равных интервалах, не превышающих `max_spacing` (по умолчанию 1000 мм).
2. Дедупликация: совпадающие концы соседних стрингеров → одна стойка (по эпсилон-расстоянию).
3. Стойки размещаются как экземпляры семейства `RF_Support`; высота берётся из параметра `RF_Floor_Height`.

---

### Алгоритм размещения плиток (05_Tiles/Place)

Файл: `lib/floor_exact.py` → `evaluate_floor_shift()` (классификация), скрипт `05_Tiles/Place/script.py` (размещение).

Для каждой ячейки сетки:

1. **Intersection** — пересечение прямоугольника ячейки с точным контуром зоны (Clipper2 boolean intersection).
2. **Классификация**:
   - **Целая** — ячейка полностью внутри контура.
   - **Простой подрез** — прямоугольная форма, 1 или 2 стороны подрезаны.
   - **Сложный подрез** (L/T/U) — 3+ стороны затронуты или вырез.
   - **Пустая** — ячейка полностью вне контура.
   - **Микрофрагмент** — подрез < 50 мм, не монтируется.
3. **Фильтрация**: минимальная ширина < 100 мм (`MIN_VIABLE_WIDTH_MM`) → пропуск.
4. **Вырезы под колонны**: определяются до 3 групп void в одной плитке. Для каждого void вычисляются: позиция (X, Y), размеры (W, H).
5. **Подбор типоразмера**: из семейства `RF_Tile` по ширине/длине/количеству void.
6. **Размещение**: экземпляр с параметрами — ряд, столбец, размеры подреза, размеры вырезов.

## 🔧 Устранение неполадок

### Частые ошибки

#### "Clipper2Lib.dll not found"

**Решение:** Убедитесь, что DLL находится в `lib/`:

```bash
ls lib/Clipper2Lib.dll
```

#### "Extension root not found"

**Решение:** Проверьте, что папка называется `.extension`:

```bash
# Должно быть: RaisedFloor.extension
```

#### "Parameter not found"

**Решение:** Запустите `00 Параметры RF` для создания параметров

#### "LoadFamily failed"

**Решение:**

1. Проверьте, что семейство загружено в проект
2. Убедитесь, что файл общих параметров задан
3. Проверьте права доступа к файлам

### Логирование

Для простых случаев можно использовать обычный logger:

```python
from pyrevit import script
logger = script.get_logger()
logger.debug("Debug message")
logger.info("Info message")
logger.error("Error message")
```

Для новых долгих, многошаговых или diagnostic-heavy команд используйте общий helper
`lib/rf_reporting.py`:

```python
from rf_reporting import ScriptReporter

reporter = ScriptReporter.from_pyrevit(
    title="Моя команда",
    log_stem="my_command",
)
reporter.stage("Старт")
reporter.write("Собираю элементы")
reporter.write("Подозрительный кейс", level="warn")
reporter.write("Ошибка записи параметра", level="error")
```

Что это даёт:

1. Одна и та же строка уходит в `pyRevit output`
2. Та же строка попадает в `script.get_logger()`
3. Пишется временный текстовый `.log` файл для разбора инцидентов

Когда использовать `ScriptReporter`:

1. GUID migration и maintenance-команды
2. smoke/diagnostic runner-ы
3. операции в несколько фаз, где важно понимать, на каком шаге всё сломалось

Когда он не нужен:

1. короткая кнопка с одним действием
2. сценарий, где достаточно одного `forms.alert(...)`

### Отчёт об ошибках

1. Окружение (Revit, pyRevit, версия плагина)
2. Сценарий использования

4. Ожидаемое vs фактическое поведение
